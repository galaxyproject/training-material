Introduction

Database schema versus Object Model

The session description is database centric

The object model is used by the code to manipulate and access the database

translation handled by an object-relational mapping SQLAlchemy

Database Platform

default out-of-the-box Galaxy installation uses SQLite

SQLite is not the recommended DBMS for running a Galaxy server production DMBS for Galaxy is PostgreSQL PostgreSQL full set of DBMS features robust support for multiple simultaneous users

What is in (and not in) the Galaxy database?

The database tracks users, groups, jobs, histories, datasets, workflows

Datasets are stored outside the database keep metadata such as data type tools themselves are not stored in the database

Understanding the Database Schema

ER diagrams and SchemaSpy

Entity-relationship diagrams understand tables and the relationships between them SchemaSpy tool for generating ER diagrams

Database conventions

Relationship tables

some tables represent relationships between things

Relationship table names contain the names of tables they are relating, suffixed with _association

Nulls and comments

PostgreSQL supports comments to table definitions but none shown in the SchemaSpy report

table definitions are generated by SQLAlchemy and SQLAlchemy does not support it
